# Section Detection Snapshot Summary

1. Recall loss vs false-positive sources: candidate recall is currently driven by the flux-based peak picker (`_generate_candidates`) plus a secondary sensitivity pass; lowering `prominence`/`sub_prominence` or reducing the `min_sec` gap increases recall but the benchmark notes show the same F1 because the NMS/score pipeline lets large flux peaks (with high repetition_break) dominate, so false positives spike before true positives increase. The baseline per-song output (e.g., `results/salami_baseline_mild_prune.full.json`) shows many kept candidates dominated by `flux_peak` contributions, while the best saver features (repetition_break, chroma_change) rarely pull sub-threshold candidates forward, linking recall loss directly to the conservative NMS+score combination.

2. Repetition pattern matching is sequence-level: `_build_ssm` aggregates beat-synchronous log-mel blocks, builds a cosine self-similarity matrix, applies Foote-style checkerboard novelty, and `_repetition_break_at_candidates` samples that checkerboard intensity at each boundary time. There is no chord-sequence hashing inside that signal, so it responds to repeated structures in the audio novelty rather than symbolic chord sequences, making it a novelty/sequence hybrid (pattern-level re-use) but not a per-chord label match.

3. Lyric density is not part of the detector path: lyrics are fetched post-hoc by the LRClib service (`third_party/ChordMiniApp/python_backend/services/lyrics/lrclib_service.py`) and stored as `{"time": seconds, "text": string}` entries when requested. No detection cache (e.g., `data/output/joker_mcgill_chords.json`) currently includes lyric frames, so lyric density can only be used as an external validation signal after detection output is produced (i.e., outside `detect_sections`).
